## On the ILC/SNAP cluster (crash-resilient layout)

The `ilcdev` dev box can die at any time; survive it by placing files by
durability, not convenience. For a project `<P>`:

- Code -> `~/clones/<P>` (home: fast, shared, stable cwd -> resumable Claude
  sessions). 20 GB quota: never commit data/models into repos.
- Pixi envs + all caches -> `/lfs/local/0/$USER/...` (node-local, fast,
  DISPOSABLE: regenerated by `pixi install` after a crash). Never put anything
  here that you cannot recreate.
- Data / inputs / outputs that must survive a crash ->
  `/dfs/user/$USER/<P>/{in,out}` (shared, large, slow).
- Temp scripts that must not pollute the repo -> `~/tmp/<P>/` (home, small).
- Large / ephemeral scratch data -> `/dfs/user/$USER/<P>/tmp/`.

Rules:
- Never write durable work to `/tmp`, `/lfs/local/0`, the repo`s `.pixi/`, or
  `$TMPDIR` -- all node-local or ephemeral and lost when the node dies.
- Pixi uses `detached-environments`, so envs live on lfs automatically; on a
  fresh node just `cd ~/clones/<P> && pixi install`.
- At the START of working on `<P>`, mint ONE uniquely-named scratch dir
  `~/tmp/<P>/<short-id>/` (and `/dfs/user/$USER/<P>/tmp/<short-id>/` for large or
  durable scratch), then use those literal paths for the whole session. Never
  share a scratch dir between concurrent sessions on the same repo.

This OVERRIDES the generic "/tmp for non-repo files" rule above when on ILC.
